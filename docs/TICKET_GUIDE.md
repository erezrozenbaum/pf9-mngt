## 1. Overview

The pf9-mngt Support Ticket System is a full-lifecycle ITSM module embedded in the platform. It covers:

- Manual tickets (service requests, incidents, change requests, inquiries)
- System-generated auto-tickets from 5 platform event hooks
- Department-based routing with optional per-user assignment
- SLA tracking with automated breach detection, Slack/Teams alerting, and auto-escalation
- Approval workflow for change gating
- Customer email notifications via 6 customizable HTML templates
- Runbook dispatch and result retrieval directly from a ticket
- Bulk operations (close stale, reassign, CSV export)
- Analytics dashboard (trend, resolution time, top assignees)
- Full RBAC integration with role-based visibility filtering

All ticket data lives in 5 PostgreSQL tables (`support_tickets`, `ticket_comments`, `ticket_sla_policies`, `ticket_email_templates`, `ticket_sequence`) created and migrated idempotently via `db/migrate_support_tickets.sql`.

## 2. Ticket Anatomy

### 2.1 Human-Readable Reference

Every ticket is assigned a unique reference on creation:

```
TKT-2026-00042
     ^^^^  ^^^^^
     year  5-digit sequence (per year, auto-incrementing)
```

Generated by `_generate_ticket_ref()` using the `ticket_sequence` table (atomic `UPDATE … RETURNING`). You can search and filter by this ref across the API and UI.

### 2.2 Ticket Types

| `ticket_type` | When to use |
|---|---|
| `service_request` | Standard user-initiated request (access, provisioning, config) |
| `incident` | Something is broken or degraded — needs urgent attention |
| `change_request` | Planned change requiring review; may gate on `requires_approval` |
| `inquiry` | Question or information request; no action implied |
| `escalation` | Ticket manually sent to another team; usually from Tier1 → Tier2 |
| `auto_incident` | System-generated incident (drift event, health score drop, runbook failure) |
| `auto_change_request` | System-generated change gate (graph delete impact, migration wave) |

### 2.3 Statuses

| `status` | Meaning |
|---|---|
| `open` | Newly created, un-assigned |
| `assigned` | Assigned to a user (set automatically on `POST /assign` or on creation when `assigned_to` is provided) |
| `in_progress` | Work has started |
| `waiting_customer` | Awaiting customer input |
| `pending_approval` | Blocked on approval gate (`requires_approval=true`) |
| `resolved` | Work complete; `resolution_note` populated |
| `closed` | Final state; ticket locked |

> **Note:** Tickets can also reach a `rejected` terminal state from `pending_approval` via `POST /{id}/reject`.

### 2.4 Priorities

| `priority` | Typical SLA response |
|---|---|
| `low` | Best effort |
| `normal` | Standard business hours SLA |
| `high` | Same-day response required |
| `critical` | Immediate response, auto-escalation on breach |

### 2.5 Routing Fields

| Field | Description |
|---|---|
| `to_dept_id` | **Required.** The department receiving the ticket. |
| `from_dept_id` | Optional. The originating department. Defaults to the caller's own department (looked up from `user_roles`). |
| `opened_by` | Set server-side from JWT `sub` claim. |
| `assigned_to` | Optional. Username of the individual assignee. If provided on `POST /tickets`, status is set to `assigned` automatically. |

### 2.6 Resource Linkage

All fields optional. Used to attach the ticket to an OpenStack resource for context:

| Field | Example |
|---|---|
| `resource_type` | `vm`, `network`, `volume`, `project` |
| `resource_id` | UUID of the resource |
| `resource_name` | Human-readable name |
| `project_id` / `project_name` | OpenStack project |
| `domain_id` / `domain_name` | OpenStack domain |

These appear in the ticket view and can be used for reporting.

### 2.7 Auto-Ticket Metadata

| Field | Description |
|---|---|
| `auto_source` | Source system: `drift`, `health_score`, `delete_impact`, `runbook_failure`, `migration` |
| `auto_source_id` | Unique identifier within that source (e.g., drift event ID, runbook execution ID) |
| `auto_blocked` | If `true`, the operation that triggered this ticket was blocked pending resolution |
| `linked_execution_id` | Runbook execution UUID |
| `linked_job_id` | Background job ID |
| `linked_migration_id` | Migration wave ID |

### 2.8 Approval Gate

When `requires_approval=true`, the ticket enters `pending_approval` status after creation and cannot be worked until an admin approves it.

| Field | Populated on |
|---|---|
| `requires_approval` | Set at creation time |
| `approved_by` / `approved_at` | `POST /{id}/approve` |
| `rejected_by` / `rejected_at` | `POST /{id}/reject` |
| `approval_note` | Note set by approver or rejecter |

### 2.9 SLA Columns

| Field | Description |
|---|---|
| `sla_response_hours` | Hours copied from matching policy at creation |
| `sla_resolve_hours` | Hours copied from matching policy at creation |
| `sla_response_at` | Deadline timestamp = `created_at + response_hours` |
| `sla_resolve_at` | Deadline timestamp = `created_at + resolve_hours` |
| `first_response_at` | Set on first non-auto comment |
| `sla_response_breached` | Set by SLA daemon |
| `sla_resolve_breached` | Set by SLA daemon |

## 3. Lifecycle & Status Flow

```
                        ┌──────────────────────────────────────────┐
                        │              CREATION                    │
                        │  POST /api/tickets                       │
                        │  POST /api/tickets/_auto (system)        │
                        └──────────────┬───────────────────────────┘
                                       │
                             assigned_to provided?
                            /                    \
                          yes                    no
                           │                     │
                      [assigned]              [open]
                           │                     │
                           └─────────┬───────────┘
                                     │
                               POST /{id}/assign
                                     │
                                 [assigned]
                                     │
                          (manual status update via PUT)
                                     │
                               [in_progress]
                                     │
                         ┌───────────┴────────────┐
                         │                        │
                  needs customer           requires approval?
                  feedback                         │
                         │                   [pending_approval]
                 [waiting_customer]          /            \
                         │           approve            reject
                         │               │                 │
                         └───────────────┘           [rejected] ← terminal
                                     │
                               POST /{id}/resolve
                                     │
                               [resolved]
                                     │
                        ┌────────────┴───────────────┐
                        │                            │
                  POST /{id}/close             POST /{id}/reopen
                        │                            │
                    [closed] ← terminal         [open] / [in_progress]
```

**Allowed re-open sources:** `resolved`, `closed`  
**Terminal states:** `closed`, `rejected` — no further transitions

## 4. RBAC & Visibility

### Permissions

| Permission | Roles granted |
|---|---|
| `tickets:read` | viewer, operator, technical, admin, superadmin |
| `tickets:write` | viewer, operator, technical, admin, superadmin |
| `tickets:admin` | admin, superadmin |

> `tickets:admin` gates: analytics, bulk actions, SLA policy CRUD, email template CRUD, `_auto` endpoint.

### Visibility Filter (`_sla_visible_filter`)

The list endpoint applies visibility rules based on the caller's role:

| Role | Sees |
|---|---|
| `admin`, `superadmin` | All tickets |
| All other roles | Tickets routed **to** their own department **OR** tickets they personally opened |

This is implemented as a SQL `WHERE` clause injected into every list/stats query.

### Internal Comments

Comments with `is_internal=true` are hidden from users with `viewer` role — they are staff-only notes. The `GET /{id}/comments` endpoint strips internal comments automatically based on role.

### 5.2 Create & Update

#### `POST /api/tickets`

Create a new ticket. Requires `tickets:write`.

**Request body (`TicketCreate`):**

```json
{
  "title":               "Cannot SSH into vm-web-01",
  "description":         "SSH connection times out since 14:00 UTC",
  "ticket_type":         "incident",
  "priority":            "high",
  "to_dept_id":          2,
  "from_dept_id":        null,
  "assigned_to":         "alice",
  "customer_name":       "Bob Smith",
  "customer_email":      "bob@example.com",
  "auto_notify_customer": true,
  "resource_type":       "vm",
  "resource_id":         "uuid-of-vm",
  "resource_name":       "vm-web-01",
  "project_id":          "openstack-project-uuid",
  "project_name":        "WebTeam",
  "domain_id":           null,
  "domain_name":         null,
  "requires_approval":   false
}
```

- `title`, `description`, `ticket_type`, `priority`, `to_dept_id` are required.
- If `assigned_to` is provided, ticket starts in `assigned` status and a confirmation email is sent to the opener.
- If `auto_notify_customer=true` and `customer_email` is set, a `ticket_created` email is dispatched.
- SLA deadlines are applied automatically from the matching policy for `(to_dept_id, ticket_type, priority)`.
- A `ticket_created` Slack/webhook notification is fired.

**Response:** `201 Created` — full ticket object including `ticket_ref`, `id`, and SLA columns.

---

#### `PUT /api/tickets/{id}`

Update any mutable fields on a ticket. Requires `tickets:write`. All fields optional.

**Request body (`TicketUpdate`):** Same fields as `TicketCreate`, all optional.

**Response:** Updated ticket object.

### 5.3 Lifecycle Actions

All lifecycle endpoints require `tickets:write` unless noted.

---

#### `POST /api/tickets/{id}/assign`

Assign (or reassign) a ticket to a user.

```json
{
  "assigned_to": "alice",
  "comment":     "Taking this one, will investigate now."
}
```

- Sets `status = assigned`, `assigned_to = alice`
- Adds an `assignment` comment to the activity thread
- Sends `ticket_assigned` email if SMTP is enabled

---

#### `POST /api/tickets/{id}/escalate`

Escalate to another department.

```json
{
  "to_dept_id": 3,
  "reason":     "Root cause is in Tier2 infrastructure configuration."
}
```

- Sets `to_dept_id`, increments `escalation_count`, records `prev_dept_id`
- Adds an `escalation` comment
- Sends `ticket_escalated` email
- Fires Slack/webhook escalation notification

---

#### `POST /api/tickets/{id}/approve`

Approve a `pending_approval` ticket. Requires `tickets:admin`.

```json
{ "note": "Approved — proceed with the change during maintenance window." }
```

- Sets `approved_by`, `approved_at`, `approval_note`
- Sets `status = in_progress`
- Adds approval comment

---

#### `POST /api/tickets/{id}/reject`

Reject a `pending_approval` ticket. Requires `tickets:admin`.

```json
{ "note": "Not enough justification provided. Please re-submit with impact analysis." }
```

- Sets `rejected_by`, `rejected_at`, `approval_note`
- Sets `status = closed` (terminal)

---

#### `POST /api/tickets/{id}/resolve`

Mark ticket resolved.

```json
{ "resolution_note": "Restarted the SSH daemon. Root cause: systemd unit file misconfigured after last kernel update." }
```

- Sets `status = resolved`, `resolved_by`, `resolved_at`, `resolution_note`
- Sends `ticket_resolved` email to customer if `auto_notify_customer=true`
- Fires resolved notification

---

#### `POST /api/tickets/{id}/reopen`

Reopen a resolved or closed ticket.

```json
{ "reason": "Issue resurfaced after 2 hours — SSH drops again intermittently." }
```

- Sets `status = open`, clears `resolved_at`
- Adds reopen comment

---

#### `POST /api/tickets/{id}/close`

Close a resolved ticket (final state).

- Requires `tickets:write`
- Sets `status = closed`, `closed_at`
- Adds close comment

### 5.11 Auto-Ticket (Internal)

#### `POST /api/tickets/_auto`

**Requires `tickets:admin`.** Intended for internal system hooks, not direct user calls. Use the `_auto_ticket()` Python helper in-process instead.

```json
{
  "title":          "Drift detected: vm-prod-01 config diverged",
  "description":    "3 properties differ from desired state.",
  "ticket_type":    "auto_incident",
  "priority":       "critical",
  "to_dept_id":     4,
  "auto_source":    "drift",
  "auto_source_id": "drift-event-uuid-abc123",
  "resource_type":  "vm",
  "resource_id":    "vm-uuid",
  "auto_blocked":   false,
  "requires_approval": false
}
```

- Idempotent: if an open ticket with the same `(auto_source, auto_source_id)` already exists, returns the existing ticket with `409` and adds a comment noting the duplicate trigger.
- `to_dept_name` may be used instead of `to_dept_id` (resolved server-side).

## 6. SLA System

### 6.1 How Policies Work

When a ticket is created, `_apply_sla(ticket_type, priority, to_dept_id, created_at)` queries `ticket_sla_policies` for the best matching row using `(to_dept_id, ticket_type, priority)`. If a match is found:

- `sla_response_hours` and `sla_resolve_hours` are copied onto the ticket
- `sla_response_at = created_at + INTERVAL 'N hours'`
- `sla_resolve_at  = created_at + INTERVAL 'N hours'`

If no policy matches, `sla_*` columns are left null and no deadlines are tracked.

### 6.2 Seeded Policies

18 policies are seeded at migration time covering the main support tiers:

| Dept | Type | Priority | Response (h) | Resolve (h) | Auto-escalate to |
|---|---|---|---|---|---|
| Tier1 Support | incident | critical | 1 | 4 | Tier2 Support |
| Tier1 Support | incident | high | 4 | 8 | Tier2 Support |
| Tier1 Support | incident | normal | 8 | 24 | — |
| Tier1 Support | service_request | normal | 8 | 48 | — |
| Tier2 Support | incident | critical | 2 | 6 | Engineering |
| Tier2 Support | escalation | critical | 4 | 24 | Engineering |
| Tier2 Support | escalation | high | 4 | 24 | Engineering |
| Tier2 Support | escalation | normal | 4 | 24 | Engineering |
| Tier2 Support | escalation | low | 4 | 24 | Engineering |
| Tier3 Support | incident | critical | 1 | 4 | Engineering |
| Engineering | change_request | high | 8 | 24 | — |
| Engineering | auto_change_request | high | 4 | 16 | — |
| Engineering | auto_incident | critical | 2 | 8 | — |
| Management | change_request | critical | 24 | 72 | — |
| Management | change_request | high | 24 | 72 | — |
| Management | change_request | normal | 24 | 72 | — |
| Management | change_request | low | 24 | 72 | — |

Add, modify, or delete policies via the `tickets:admin` API or directly in SQL.

### 6.3 SLA Daemon

An asyncio background task running inside the API process on a **15-minute interval**. On each tick it:

1. Queries all non-closed/non-resolved tickets where `sla_response_at < now()` or `sla_resolve_at < now()`
2. Sets `sla_response_breached = true` / `sla_resolve_breached = true` as appropriate
3. Sends `ticket_sla_breach` email to the assigned user (if `SMTP_ENABLED`)
4. Posts a Slack/Teams webhook notification tagging the ticket ref
5. Inserts a `sla_breach` comment in the ticket activity thread (with `is_internal=true`)
6. If the matching SLA policy has `auto_escalate_on_breach=true` and an `escalate_to_dept_id`, automatically calls the escalate logic — re-routing the ticket to the target department

> **Note:** The daemon sets breach flags only once. Once `sla_response_breached=true`, the response deadline check is skipped on subsequent ticks to avoid duplicate notifications.

## 8. Auto-Ticket System

### 8.1 Trigger Sources

Five platform hooks automatically create tickets when significant events occur:

| Source | `auto_source` | File | Condition |
|---|---|---|---|
| Drift detection | `drift` | `db_writer.py` | Drift event with severity `critical` or `warning` is written |
| Health score | `health_score` | `graph_routes.py` | A node's `health_score` drops below **40** |
| Delete gate | `delete_impact` | `graph_routes.py` | `POST /api/graph/request-delete` — impact analysis non-empty |
| Runbook failure | `runbook_failure` | `runbook_routes.py` | Runbook execution raises an unhandled exception |
| Migration wave | `migration` | `migration_routes.py` | Wave status transitions to `complete` |

All five hooks call `_auto_ticket()` internally via a direct in-process function call (no HTTP round-trip).

### 8.2 Deduplication Logic

Before creating a new auto-ticket, the system checks:

```sql
SELECT id FROM support_tickets
WHERE  auto_source    = $1
  AND  auto_source_id = $2
  AND  status NOT IN ('closed', 'resolved', 'rejected')
LIMIT 1
```

- **If a match exists:** The existing ticket is returned, and a `system` comment is appended noting the repeated trigger. No new ticket is created.
- **If no match:** A new ticket is created with a fresh `ticket_ref`.

This prevents duplicate ticket spam from continuously firing events (e.g., a VM whose health score stays below 40 for an extended period).

### 8.3 Using `_auto_ticket()` in Code

Import and call from any API module:

```python
from ticket_routes import _auto_ticket

ticket = await _auto_ticket(db, {
    "title":          "Drift detected on vm-prod-01",
    "description":    "3 config properties diverged from desired state.",
    "ticket_type":    "auto_incident",
    "priority":       "critical",
    "to_dept_id":     4,          # Tier3 Support
    "auto_source":    "drift",
    "auto_source_id": str(drift_event.id),
    "resource_type":  "vm",
    "resource_id":    vm_id,
    "resource_name":  vm_name,
    "auto_blocked":   True,
    "requires_approval": False
})
```

Returns the ticket dict (new or existing). If `auto_blocked=True`, set a flag in your calling code to block the triggering operation until `status` reaches `resolved` or `closed`.

## 9. Runbook Integration

Tickets and runbooks are linked in two directions:

**Ticket → Runbook (dispatch):**
`POST /api/tickets/{id}/trigger-runbook` dispatches a runbook with the provided `runbook_name` and `parameters`. The `linked_execution_id` is stored on the ticket, and a `runbook_result` comment records the dispatch.

**Runbook → Ticket (auto-created):**
If a runbook execution fails, `runbook_routes.py` calls `_auto_ticket()` with `auto_source=runbook_failure` and sets `linked_execution_id` on the new ticket. This allows ops teams to track all runbook failures through the ticket queue.

**Retrieving results:**
`GET /api/tickets/{id}/runbook-result` returns the latest execution document for `linked_execution_id` from the runbook engine. Poll this after dispatch or await a webhook completion event.

## 12. Admin Operations

### 12.1 Common SQL Queries

**All open tickets for a department:**
```sql
SELECT ticket_ref, title, priority, status, assigned_to, created_at
FROM   support_tickets
WHERE  to_dept_id = 2  -- Tier2 Support
  AND  status NOT IN ('closed', 'resolved', 'rejected')
ORDER  BY created_at DESC;
```

**SLA-breached tickets still open:**
```sql
SELECT ticket_ref, title, priority, to_dept_id, assigned_to,
       sla_response_at, sla_resolve_at
FROM   support_tickets
WHERE  (sla_response_breached = true OR sla_resolve_breached = true)
  AND  status NOT IN ('closed', 'resolved', 'rejected')
ORDER  BY sla_resolve_at ASC NULLS LAST;
```

**Ticket volume by type this month:**
```sql
SELECT ticket_type, COUNT(*) AS cnt
FROM   support_tickets
WHERE  created_at >= date_trunc('month', now())
GROUP  BY ticket_type
ORDER  BY cnt DESC;
```

**Auto-tickets by source (last 30 days):**
```sql
SELECT auto_source, COUNT(*) AS cnt, AVG(EXTRACT(EPOCH FROM (resolved_at - created_at))/3600) AS avg_resolve_h
FROM   support_tickets
WHERE  auto_source IS NOT NULL
  AND  created_at >= now() - INTERVAL '30 days'
GROUP  BY auto_source;
```

**Activity thread for a ticket:**
```sql
SELECT c.author, c.comment_type, c.body, c.is_internal, c.created_at
FROM   ticket_comments c
JOIN   support_tickets t ON t.id = c.ticket_id
WHERE  t.ticket_ref = 'TKT-2026-00042'
ORDER  BY c.created_at ASC;
```

### 12.2 Managing SLA Policies

Add a new policy via API:
```http
POST /api/tickets/sla-policies
Authorization: Bearer <admin-token>

{
  "to_dept_id": 3,
  "ticket_type": "service_request",
  "priority": "high",
  "response_sla_hours": 4,
  "resolution_sla_hours": 16,
  "auto_escalate_on_breach": false
}
```

Or via SQL:
```sql
INSERT INTO ticket_sla_policies
  (to_dept_id, ticket_type, priority, response_sla_hours, resolution_sla_hours, auto_escalate_on_breach)
VALUES
  (3, 'service_request', 'high', 4, 16, false)
ON CONFLICT (to_dept_id, ticket_type, priority)
DO UPDATE SET response_sla_hours = EXCLUDED.response_sla_hours,
              resolution_sla_hours = EXCLUDED.resolution_sla_hours;
```

> Policy changes only affect **new** tickets. To reclaculate deadlines on existing tickets, run a manual UPDATE on `sla_response_at` / `sla_resolve_at`.

## 13. Deployment & Migration

The ticket system schema is fully managed by:

```
db/migrate_support_tickets.sql
```

This script is **idempotent** — safe to run multiple times. All `CREATE TABLE` statements use `IF NOT EXISTS` and all `INSERT` seeds use `ON CONFLICT DO NOTHING`.

**Run migration manually:**
```bash
docker exec pf9_db psql -U pf9 pf9_mgmt -f /docker-entrypoint-initdb.d/migrate_support_tickets.sql
```

**Auto-migration on startup:**
`_ensure_tables()` in `ticket_routes.py` runs the migration file via `asyncpg` on first import. This means the tables and seed data are guaranteed to exist whenever the API starts, even on a fresh database.

## 14. Error Reference

| HTTP status | Cause |
|---|---|
| `400 Bad Request` | Invalid field values; lifecycle action not valid for current status |
| `401 Unauthorized` | Missing or expired JWT |
| `403 Forbidden` | Insufficient role (e.g., non-admin calling analytics) |
| `404 Not Found` | Ticket ID does not exist, or template name not found |
| `409 Conflict` | Duplicate SLA policy `(dept, type, priority)`; duplicate auto-ticket with same `(auto_source, auto_source_id)` |
| `422 Unprocessable Entity` | Pydantic validation failure (missing required field, wrong type) |
| `500 Internal Server Error` | Database error; SMTP failure (logged, email skipped) |

> SMTP failures do **not** roll back the ticket operation. The ticket action succeeds; the email failure is logged with `ERROR` level. Check `api/logs/` for SMTP trace.
